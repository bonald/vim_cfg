Section 3: Supported Device

Device

DESCRIPTION

New Added

CTC5160

CTC5160 is a high feature, high performance IP/Ethernet all-in-one switch designed to fulfill next generation Enterprise, Carrier Ethernet and Packet Transport access/aggregation network requirements. With integrated Layer2 to Layer4 packet processing engine, advanced traffic manager and on-chip OAM engine, it provides rich feature set at up to 120Gbps performance

N

CTC8096

CTC8096 is designed for cost effective high density 10GbE/40GbE applications with unique 4x100GbE uplink capability. It incorporates innovative CloudWise™ technology providing cutting-edge silicon level intelligence for visibility, resilience, efficiency, flexibility and manageability C the key building blocks of Cloud 3.0 applications

N

CTC7148

CTC7148 is a high-performance switching device with converged speed/ service/ application/ X-engine for next generation of multi-gig applications with 10G/25G/40G/50G/100G uplink capabilities to meet services and applications convergence requirements of flat network.

N

Section 6: Compatibility

Section 6.1: ACL&SCL field mode

SDK provide new mode for ACL&SCL entry compared with v3.7.3RC, also is compatible with old mode used in V3.7.3RC and the version earlier than V3.7.3RC.
Add new variable mode in struct ctc_acl_entry_t&ctc_scl_entry_t, which means compatible mode or new mode.

6.1.1. New mode

In this mode, must set mode in stuct ctc_acl_entry_t or ctc_scl_entry_t to 1.Here take acl module for example to describe how to use this mode:

1.    Call API ctc_acl_create_group(uint32 group_id, ctc_acl_group_info_t* group_info)

2.    Call API ctc_ acl_add_entry(uint32 group_id, ctc_acl_entry_t* acl_entry)

Notice: Compared with compatible mode, in this mode, do not need provide key&action field, just provide key_type, entry_id, priority, hash_field_sel_id(for hash used), also variable mode must set to 1

3.    Call API ctc_acl_add_key_field(uint32 entry_id, ctc_field_key_t* key_field)

Notice: User can call API ctc_acl_add_key_field to add key field separately for added entry. In the step, user can continually call API to add needed key field until all key field completely added

4.    Call API ctc_acl_add_action_field(uint32 entry_id, ctc_acl_field_action_t* action_field)

Notice: User can call API ctc_acl_add_action_field to add action field separately for added entry. In the step, user can continually call API to add needed key field until all action field completely added

5.    Call API ctc_acl_install_entry or ctc_acl_install_group to install entry

This mode provide more flexible used for add key&action field, and more extended for future used. Also, this mode is fit for SCL module

a.    About how to select field in ctc_field.h for every key_type, please refer to ACL Application document

b.    About how to use acl detail in this mode, please refer to SdkTypicalConfiguration.chm

6.1.2. Compatible mode

In this mode, must set variable mode in stuct ctc_acl_entry_t to 0

1.    Call API ctc_acl_create_group(uint32 group_id, ctc_acl_group_info_t* group_info)             

2.    Call API ctc_ acl_add_entry(uint32 group_id, ctc_acl_entry_t* acl_entry)

Notice: Must provide all key in parameter struct key in acl_entry, provide all action in parameter struct action in acl_entry, also variable mode must set to 0

3.    Call API ctc_acl_install_entry or ctc_acl_install_group to install entry

Notice: This mode only support several types of key_type, new key_type no longer to add to the mode
Compatible key type:

1.    CTC_ACL_KEY_MAC

2.    CTC_ACL_KEY_IPV4

3.    CTC_ACL_KEY_IPV6

4.    CTC_ACL_KEY_HASH_MAC

5.    CTC_ACL_KEY_HASH_IPV4

Also in this mode only support key field and action field supported by CTC8096&CTC5160, new added field such as forwarding key is not support.

Section 6.3: Dynamic switch

SDK provide new API to do dynamic switch compared with v3.7.3RC, also old API for dynamic switch is also supported.

6.3.1. New mode

Provide new API to do dynamic switch based on port, new API is :
int32 ctc_port_set_interface_mode(uint8 lchip, uint32 gport, ctc_port_if_mode_t* if_mode)

6.3.2. Compatible mode

In v3.7.3RC or earlier version, dynamic switch is based on serdes id, API is:
int32 ctc_chip_set_serdes_mode(uint8 lchip, ctc_chip_serdes_info_t* p_serdes_info)
Before using this API, user must get gport and serdes id mapping relationship, convert gport to serdes id, and do dynamic switch. Also, switch by serdes id will make SDK process complexly in some case, for example, 40G & 100G port, will have 4 serdes but only one mac and pcs.

Section 6.4: Qos

6.4.1 Classify

1.   In V3.7.3RC or earlier version, SDK provide cos/dscp/exp can only from port property using CTC_PORT_DIR_PROP_QOS_DOMAIN in ctc_port_direction_property_t. In new SDK provide cos domain/dscp domain/exp domain separately for CTC7148, and can get from port,l3if and nexthop

a)    For bridge packet, qos domain is from port, also SDK provide CTC_PORT_DIR_PROP_QOS_COS_DOMAIN&CTC_PORT_DIR_PROP_QOS_DSCP_DOMAIN separately

b)    For route packet, qos domain is from l3if

c)    For mpls packet, qos domain is from dsmpls

2.   In new version, SDK add new property CTC_MPLS_ILM_QOS_MAP and new struct ctc_mpls_ilm_qos_map_t to do qos map, in V3.7.3RC or earlier version using CTC_MPLS_ILM_LLSP

3.   For CTC7148, internal priority has 16 levels. For CTC8096, internal priority has 64 levels

6.4.2 Shaping

In new version SDK, for CTC7148, dma channel support total shaping for all dma channel, also support per DMA channel shaping. For CTC8096, only support per dma channel shaping

6.4.3 Policer

In hierarchical metering, for CTC7148, the color of per level meter is completely independent. For CTC8096, the color of per level meter can be configured to independent or not

6.4.4 Resource Manage

For CTC7148, the congestion level of resource manager has 4 levels. For CTC8096, the congestion level of resource manager has 8 levels

Section 6.5: IPFIX

In v3.7.3RC or earlier version, for CTC8096 IPFIX is enabled by port, using API:
ctc_ipfix_set_port_cfg(uint32 gport,ctc_ipfix_port_cfg_t* ipfix_cfg)
For CTC7148, IPFIX is enabled by acl, so the API above is not used, using below API instead:

Add Acl entry, and action is ipfix, for example:

Section 6.6: ARP

The SDK provides Create/ Update/Remove ARP API to manage ARP resources,benefits are as follows:

 1.  IPUC/IP-TUNNEL/MPLS can share ARP resources

2.  When the ARP changes, only need to update Arp, do not need to refresh each Nexthop

6.6.1. New mode

1.    Create ARP using API: extern int32 ctc_nh_add_nexthop_mac(uint16 arp_id, ctc_nh_nexthop_mac_param_t* p_param)

2.    Create nexthop using ARP ID from step 1

3.    When arp update, just update ARP ID from step 1 using API:int32 ctc_nh_update_nexthop_mac(uint16 arp_id, ctc_nh_nexthop_mac_param_t* p_new_param)

4.    Nexthop created from step 2 do not have to change, after step 3, all nexthop using ARP ID from step 1 will update

6.6.2. Compatible mode

Just take ipuc nexthop for example.

1.    Using API to create ipuc nexthop: int32 ctc_nh_add_ipuc(uint32 nhid, ctc_ip_nh_param_t* p_nh_param)

2.    Arp information is set id parameter p_nh_param->oif, which is for vlan and dest port

3.    Update nexthop when arp changed, using API: int32 ctc_nh_update_ipuc(uint32 nhid, ctc_ip_nh_param_t* p_nh_param). If arp and nexthop have N:1 mapping, must update every nexthop using the same arp.

Section 6.7: Stacking

In new version, SDK provide new stacking header format, which used for reduce payload of stacking header, this feature only support CTC7148 interconnect with itself or subsequent chip, when sdk init, version should be set CTC_STACKING_VERSION_2_0

struct ctc_stacking_glb_cfg_s

{

    ...

    uint8  version;  /**< [D2] stacking version, refer to ctc_stacking_version_t*/

    ...

};

typedef struct ctc_stacking_glb_cfg_s ctc_stacking_glb_cfg_t;

If CTC7148 interconnect with CTC5160 or CTC8096, version should be set CTC_STACKING_VERSION_1_0, stacking header will use old static 32 byte data
